## Context

Play-test 4 lands on the M4.3 build: the actionability signal is the **coloured
drop-shadow bar** (`HudChrome` `BLEED*`), the CRT chrome is retired, the shard rail is
a vertical rail left of the icon column, and the sequence tool is the `HudQueue`
draining-ghost strip. **The board is *correct*** — the log confirmed strictness holds
and the scoring rules fire. Unlike M4.1 (the Imp-Lord-ID class of bug), **nothing here
is a scoring or data defect**. It is all legibility, juice, and one real sequence-render
bug. Five workstreams:

- **A — the cue bar** *(feedback 1, 2, 5)*. Lock the name (**cue bar / `CUE`**, chosen
  over "AHB" to dodge the keybind-*hint* collision), restore the **width axis** (colour
  + width redundant per [X1]), and give **Tyrant a max-width override** so the burst
  go-signal is the widest thing on the board even in SOON.
- **B — the shard rail** *(feedback 3)*. Recut the rail as **5 large dots**, three
  explicit states (empty / incoming / filled), centred on the **Essential** viewer's
  full height.
- **C — the sequence tool** *(feedback 4, 7)*. Fix the real bug — the strip stops
  brightening the current step after step 1 — and add **cast-start** feedback alongside
  the existing succeeded-pop.
- **D — cast-start flair on the icon** *(feedback 6)*. A quiet flash next to a casting
  ability's icon.
- **E — burst confirmation text** *(feedback 8)*. The ability name floats up over the
  character *after* a burst-window press — positive reward for landing the burst, and
  the loudest member of the same cast-feedback family as C and D.

*(Feedback item 0 — keybinds — is **closed**: a cold-scan timing gap, warm now, no fix.
Imp Lord resolves via its real bar id `1276452`. Feedback item 9 — tooling — is
**M4.5**, see `m4.5-plan.md`.)*

## Approach, per workstream

### A — the cue bar: name, width axis, burst emphasis

**A1 — rename `bleed`/`BLEED*` → `cue`/`CUE`.** A clean mechanical rename inside
`HudChrome.lua` plus the two public method names and the docs:

- Symbols: `BLEED` table → `CUE`; `BLEED_THICK` → `CUE_MIN` (it is now the *minimum*
  width); `paintBleed` → `paintCue`; `ensureBleed` → `ensureCue`; `anchorBleed` →
  `anchorCue`; `o.bleed` → `o.cue`; `o.bleedLevel` → `o.cueLevel`; the header comment
  block (`HudChrome.lua:110`) rewritten in the new vocabulary. `H.DOT_COLORS`'s
  derivation from `BLEED.*.c` follows the rename.
- **Public methods too:** `H.SetDot` → `H.SetCue`, `H.GetDot` → `H.GetCue`. Only three
  call sites (`HudChrome.lua:744`, `HudState.lua:651`, `:665`) — leaving a bar named
  `SetDot` is a lasting wart, and the churn is trivial. (`H.DOT_COLORS` can keep its
  name — it is read by `HudRow.levelTag` as the level→word map; rename only if the row
  code is already being touched.)
- Docs: `guidance-model.md §0.5.8.7` and `spec.md §3` gain "the **cue bar** — the
  level signal" terminology.

Do A1 first so the rest of the diff reads in the settled vocabulary.

**A2 — restore the WIDTH axis.** Width becomes a per-level field on the `CUE` table,
with the current thickness as the **minimum**:

| Level | Colour (unchanged) | Width | Reads as |
|---|---|---|---|
| `JUDGE` (Implosion up — your call) | cyan | `CUE_MIN` = 14 | ready, discretionary |
| `SOON` (anticipation on NEVER) | yellow | 14 | coming up |
| `ROTATION` | green | 22 | **press this now** (fatter) |
| `LATE` | green | 24 | overdue (fattest sustain) |
| **burst-emphasis override** (A3) | (its level's hue) | `CUE_BURST` = 30 | the widest bar on the board |

```lua
local CUE_MIN, CUE_BURST = 14, 30
local CUE = {
  JUDGE    = { c = {0.27,0.88,1.00}, a = 0.85, w = CUE_MIN },
  SOON     = { c = {1.00,0.86,0.15}, a = 0.80, pulse = 0.85, w = CUE_MIN },
  ROTATION = { c = {0.30,1.00,0.48}, a = 1.00,               w = 22 },
  LATE     = { c = {0.42,1.00,0.58}, a = 1.00, pulse = 1.3,  w = 24 },
}
```

`anchorCue` reads `spec.w` in place of the constant `t:SetWidth(BLEED_THICK)`
(`HudChrome.lua:366`); `paintCue`'s gradient math is width-agnostic. Recompute
`H.ROW_OFFSET` off the **widest possible** bar (`CUE_BURST`, not `CUE_MIN`) so the debug
row never overlaps a Tyrant bar (`H.ROW_OFFSET = CUE_BURST + 30`).

**A3 — Tyrant is the widest cue, even in SOON.** Width comes from an **emphasis** bit,
not the level, so an incoming Tyrant beats a ROTATION sustain press. ⚠ **`goGate` is the
wrong bit** — it is Tyrant *and* Dreadstalkers (`SpecDemonology.lua:30`), and feedback 5
wants *Tyrant specifically* the widest. So add a **dedicated per-ability field**
`emphasis = "burst"` on Tyrant's spec entry (`SpecDemonology.lua:118`), extensible to a
lesser tier later. Flow:

- `HudScore.For` copies it through: `out.emphasis = info.emphasis` (nil for everything
  but Tyrant).
- The score→cue bridge passes it as a 5th arg: `H.SetCue(item, viewer, level,
  judgeReady, emphasis)` (`HudState.lua:651`).
- `SetCue` width: `local w = (emphasis == "burst") and CUE_BURST or (spec.w or
  CUE_MIN)` — applied regardless of level, so a **SOON** (yellow) Tyrant is still the
  widest bar. Hue keeps carrying level; width now carries "this is the burst go-signal."

### B — the shard rail: 5 dots, 3 states, Essential-height

`buildRail` today draws `cap` (`ns.SHARD_CAP or 5`, `:984`) thin vertical bar-segments
over the measured viewer span, each with a `fill` and a 4-edge in-flight marker
(`:1020`). Recut it to read as **dots**, with the three states made explicit:

- **5 large dots, more gap.** Keep the slim rail width (`RAIL_SEG_W` ~12–14) but make
  each segment a **round-ish dot**: a masked circle where the mask texture is available
  (same `pcall`-guarded `SetMask` idiom the retired disc used), square fallback. Raise
  `RAIL_GAP` so five read as five discrete dots, not a sliced bar. Dot size derives from
  the span: `segH = (span - (cap-1)*RAIL_GAP) / cap` (already the formula at `:993`),
  clamped so a dot is roughly as tall as it is wide.
- **Three explicit states** (one `paintDot(i, state)` driven off shard count + the
  in-flight projection):
  - **empty** — `RAIL_EMPTY` ring outline, no fill (a placeholder circle).
  - **incoming** — a shard a live cast will add (the existing in-flight projection),
    drawn as a **dim/dashed ring** so it reads as *projected*, matching the
    hollow-estimate convention the cue bar and napkin already use. This is the one new
    visual state; the projection data already exists (spend-side START tracking).
  - **filled** — solid `RAIL_COL` fill (mode-tinted as today).
- **Centre on the Essential viewer's full height.** Confirm the rail measures and
  anchors to the **Essential** viewer specifically (`f:SetPoint("BOTTOMRIGHT", viewer,
  "BOTTOMLEFT", …)`, `:1000`) and that the five dots are vertically centred over that
  span rather than bottom-aligned to a 3-icon fallback.

⚠ The **cap animation** (flash + sparks on the sibling frame, `:1044`) is **untouched** —
already M6-fenced and rate-limited. This is dot geometry + the incoming-state ring only.

### C — the sequence tool: brighten the live step + cast-start feedback

**C1 — the real bug.** `HudQueue.lua:191` is:

```lua
local col = (inst.ready and i == cur) and BRIGHT or DIM
```

`ready` is the consumer's prereq wall (`SetReady`, `:279`). Once you cast the opener's
step 1, shards drop below the 5-shard prereq → the consumer flips `ready=false` → the
whole strip dims and the new current step never brightens. **The dim-until-ready gate
must only bite while `primed`** (before the sequence has started). Once un-primed, always
brighten `cur`:

```lua
local lit = (i == cur) and (inst.primed == false or inst.ready)
local col = lit and BRIGHT or DIM
```

- **Opener** arms `primed = true` (dim until the wall is down); the first matching press
  un-primes it (`:332`), and from then on the current step always brightens. ✔ the fix.
- **Burst** re-arms un-primed (`:260`), so its current step brightens from the start —
  which is the desired "always highlight the next press" behaviour.

One-line predicate change, no new state (`primed` already exists, `:272`).

**C2 — cast feedback on START as well as SUCCEEDED.** `playPop` (`:291`) fires only on
succeeded/advance. Add a **distinct** start cue:

- New `QueueMeta:playCastStart(spellID)` — if `spellID` matches the current step's
  ability, apply a brief **"casting…" shimmer** to that cell (a short alpha
  in-out on the strip's current-cell colour), visibly different from the
  pop-and-rise of an advance. A `STOP` without a `SUCCEEDED` (fizzle / interrupt) clears
  the shimmer without advancing.
- Wiring: `HudState`'s `UNIT_SPELLCAST_START` branch (`:996`) routes to
  `HudOpener.OnCastStart(spellID)` / `HudBurst.OnCastStart(spellID)` — the start-side
  siblings of the existing `OnCast` (`:1004–1008`) — which forward to the owned pane's
  `playCastStart`.

### D — cast-start flair on the icon

A small, quiet flash by a casting ability's icon on `UNIT_SPELLCAST_START`:

- New `H.CastFlash(item)` — a one-shot fade primitive that **reuses the cue animation
  plumbing** (pooled, no new animation group), a brief low-alpha bloom on the chrome
  frame beside the icon.
- Driven from the same `HudState` START branch (`:996`): resolve `spellID → item` via
  the registry, call `H.CastFlash(item)`.
- **Cast-time spells only** — an instant is already confirmed by the queue advance and a
  flash on every GCD would be noise. ⚠ The icon already carries the cue bar + keybind +
  native swipe; this is *confirmation the cast is going*, never a call. Keep it brief and
  low-alpha; if it competes with the cue bar in play, drop it to burst-window casts.

### E — burst confirmation text (floating over the character)

Positive reward for landing a burst-window press — the name floats up over the character
after the cast. Because it fires **after** the press and only echoes what you *did*, it
carries no instruction; it is the loudest member of the C/D cast-feedback family and the
payoff for A3's "make BURST shout."

- **New module `HudFloat.lua`** — a pool of rising/fading `FontString`s. `HudFloat.Say(text, opts)`
  spawns one at the player anchor, animates it up (~40px over ~1.2s) while fading, then
  returns it to the pool. M1 already proved pooled-FontString animation, so this is
  known-feasible; it reuses that idiom.
- **Anchor:** the player **nameplate** when shown (`C_NamePlate.GetNamePlateForUnit("player")`),
  falling back to a **fixed screen anchor** above centre when the personal nameplate is
  off. (Personal-resource nameplate availability varies by setting — the fallback keeps
  it working regardless.)
- **Trigger:** `HudBurst.OnCast(spellID)` (a burst-window press succeeded) →
  `HudFloat.Say(abilityName)`. **Scoped to burst steps** (`ns.SpecBurst` membership /
  `burstAlign`), not every cast, so it stays a burst celebration, not a running combat
  log. **Tyrant reads loudest** (bigger / brighter treatment than HoG/Demonbolt/Implosion
  fill presses), mirroring A3.
- **Feel is the tunable** (size, rise distance, fade curve, Tyrant emphasis). No ethos
  gate — the only watch-item is keeping it burst-scoped so it doesn't clutter.
- Audio stays **M6**: this is visual-only; the `HudChrome` no-`PlaySound` fence is
  untouched.

## Build order

Sequential, each independently verifiable at a dummy; A–E ship as **one** release
(v0.24.0). No heavy phase ceremony — this is a tuning pass.

1. **A — cue bar** (rename → width axis → Tyrant override → `ROW_OFFSET`). The headline;
   do the rename first so everything downstream reads in the new vocabulary.
2. **B — shard dots** (5 dots, three states, Essential-centred).
3. **C — sequence** (C1 the brighten-current fix first — it's the real bug — then C2 the
   start shimmer).
4. **D — cast-start flair** (the icon flash; shares the START wiring with C2).
5. **E — burst confirmation text** (the `HudFloat` module; shares the burst `OnCast` with
   the existing route).

Then luaparser-check, bump `.toc`, `wowkb.addon release cdmp --minor`, `/reload`,
re-verify below. (M4.5's tooling — luacheck + busted — lands separately and would then
gate future releases; see `m4.5-plan.md`.)

## Verification (in-game, at a target dummy)

1. **Cue width (A):** ROTATION/LATE bars are visibly fatter than SOON/JUDGE; an incoming
   **Tyrant** is the single widest bar, even at SOON (yellow + widest); colour still
   carries level; the debug row never overlaps a Tyrant bar.
2. **Shard dots (B):** five large dots centred on the **Essential** stack; empty dots
   read as placeholder rings, a filled dot is solid, and starting a generator cast shows
   an **incoming** (projected) ring before the shard lands.
3. **Sequence brighten (C1):** opener on, cast step 1 → the strip **brightens step 2**
   (not dim); each consumed step drops and the new current always lights.
4. **Sequence cast cue (C2):** a cast **start** shows a distinct "casting…" shimmer;
   **succeeded** still pops-and-rises; a fizzle clears the shimmer without advancing.
5. **Icon flair (D):** casting a cast-time ability (HoG / Demonbolt) flashes by its icon,
   briefly and quietly; instants don't.
6. **Burst text (E):** landing a burst-window press floats its name up over the
   character; Tyrant reads loudest; non-burst casts don't float.
7. **`/cdmp hud log`** — the recorder captures the pass; read the `queue` events for
   brighten/advance timing and tune next session (M3e doctrine — not this one).

## Open questions / risks

- **Width axis vs. row text (A).** Fatter bars push `H.ROW_OFFSET`; confirm at 1440p+
  that the debug row clears the widest (Tyrant) bar. `ROW_OFFSET = CUE_BURST + 30` covers
  it by construction.
- **`emphasis` is a data bit, not a spellID (A3).** Kept in `SpecDemonology` so a second
  spec (M7) inherits it and nobody re-hardcodes "Tyrant." Deliberately a **new field**,
  not `goGate` (which is Tyrant + Dreadstalkers).
- **Incoming-dot readability (B).** The projected ring must read as *not-yet-held*
  without looking like a filled dot; if it reads ambiguous, dim it further or dash it,
  don't brighten it (matching the hollow-estimate convention).
- **Cast-start flair noise (D).** The icon is busy; if the flash competes with the cue
  bar, drop to alpha-only or gate to burst-window casts. It is confirmation, never a
  call.
- **Float anchor (E).** Personal-nameplate availability varies by setting; the fixed
  screen-anchor fallback keeps it working. Watch that it doesn't collide with Blizzard's
  own FCT stream — if it does, offset the anchor rather than fighting it.
- **Audio stays M6.** All juice this pass is visual; the no-`PlaySound` fence and the
  `onCorrect/onMiss/onComplete` seams are untouched.
