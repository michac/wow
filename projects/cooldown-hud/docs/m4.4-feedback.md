## 1. Name the shadow-bar widget (terminology lock)

The coloured edge bar reads well now. It needs a stable name in code + docs. Player
proposes **"ability hint bar" (AHB)**. Open to a better name.

- Candidate: **AHB / Ability Hint Bar** — "hint" fits inform-don't-instruct, but
  collides slightly with the *keybind* hint (also a "hint").
- Alternative to weigh: **"cue bar"** / **"action cue"** / **"edge cue"** —
  distinct from the keybind hint, still non-instructive.
- Decision to make in the plan, then **document it** (spec.md/guidance-model +
  rename the `bleed`/`BLEED*` symbols in `HudChrome`).

## 2. AHB: restore the WIDTH axis (two axes = colour + width)

It was helpful when **discretionary + soon** abilities were **thinner** and
**recommended (ROTATION)** were **wider**. Restore that:

- **Current width = the MINIMUM.** JUDGE (discretion) + SOON stay at the current
  thickness.
- **ROTATION / LATE = WIDER.** Recommended presses get a fatter bar.
- So the bar carries level on **hue AND width** redundantly ([X1]).

## 3. Shard rail: compress to 5 large dots

The rail is still too busy. Redesign:

- **5 large dots** (one per shard), stacked **vertically**.
- Each dot has **3 states**: **empty** (placeholder outline), **incoming** (a shard
  in flight / projected), **filled** (held shard).
- Keep it **flowing vertically**, and **centre it on the full vertical height of the
  Essential cooldown bar** (not the bottom-N-icons span it uses now).

## 4. Sequence panel: highlight the NEXT key after the first

The panel **drops** consumed steps but does **not brighten the new current step**
after step 1. It should always highlight the current (next-to-press) step.

- Likely cause: the M4.3 "dim until prereqs met" gate keys on `inst.ready`, and once
  you cast the opener's first ability your **shards drop below 5**, so the 5-shard
  prereq un-meets → `ready = false` → nothing brightens. The dim-until-ready gate
  should apply **only before the sequence has started** (while primed); once
  started (un-primed), always brighten the current step.

## 5. Draw more attention to BURST

- Make **Tyrant specifically the WIDEST AHB of all**, even in **SOON** mode — so an
  incoming Tyrant window shouts louder than any sustain press. A per-ability width
  emphasis / override for the burst go-signal.

## 6. Cast-start flair on the icon

- When an ability's **cast STARTS** (`UNIT_SPELLCAST_START`), show a **little flair**
  next to that cooldown icon — quick visual feedback that the cast is going.

## 7. Sequence bar: cast feedback on start AND succeeded

- The sequence strip should give visual feedback on **cast start** as well as **cast
  succeeded** (today `playPop` only fires on succeeded/advance).

## 8. Floating combat text over the character?

- Can the addon show **floating combat text over the character** (custom rising/
  fading text)? If so, **surface the burst casts** that way — announce the burst
  window's presses over the head.
- Feasibility to establish in the plan (custom animating FontString anchored to the
  player nameplate / a screen anchor; Blizzard FCT vs our own).

## 9. Tooling: static analysis + unit tests

Investigate what we can accomplish, **both locally and in the WoW client**:

- **Static analysis (local/CI):** we already luaparser-check syntax; add **luacheck**
  with a WoW-globals config (undefined globals, unused vars), and consider **LuaLS**
  with WoW API type definitions.
- **Unit tests (local):** **busted** + a WoW API mock. The **pure-logic modules are
  the prize** — `HudScore` (pure function of readable state), `HudQueue` queue
  mechanics (drop-through / count / skip), `HudNapkin` — all testable with stubbed
  `ns.*`. Frame-drawing (`HudChrome`) is harder (needs `CreateFrame` stubs).
- **In-client testing:** a `/cdmp selftest` that runs assertions against known states
  and prints pass/fail — the only way to exercise the live Secret-Value / CDM API.
