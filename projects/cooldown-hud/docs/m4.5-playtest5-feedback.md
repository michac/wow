## 0. What went wrong on the first attempt (resolved before you re-run)

The 15:00 capture flushed `CDMProbeDB.probe = {}` — an **empty** structured store,
with the newest report inside it stamped `version 0.22.0/0.23.0`. Cause: the probe
ran under pre-0.25.0 code. ghaddons had written v0.25.0 to disk, but WoW only
re-executes addon Lua on **`/reload`**, and the `/reload` that flushed the file
happened *before* the new code was in memory — so the run collected old-format data
and the reload persisted it.

`wowkb.cdmp check` reported **"No structured probe capture"** and exited without
asserting anything. That is the designed behaviour and worth noting as a small win:
a missing observation read as *not covered*, never as a pass.

**Therefore the protocol below opens with a `/reload`.** Verify once, before
probing, that `/cdmp` reports **0.25.0** — if it doesn't, nothing else in this doc
is meaningful.

## 1. The capture protocol (run this, then fill in §2+)

```
/reload                  ← loads v0.25.0.  NOT optional, and it goes FIRST.
/cdmp                    ← confirm it says 0.25.0
/cdmp probe clear        ← resets counters AND stored snapshots → coverage is THIS session's
/cdmp probe              ← out of combat, at the dummy
/cdmp probe guide        ← see what's still missing
<pull the dummy — get a Demonic Core proc, a Tyrant, imps out>
/cdmp probe              ← IN COMBAT, mid-pull
/cdmp probe guide        ← re-type to re-check; aim for "coverage complete"
/reload                  ← flushes SavedVariables.  Also not optional.
```

Then, locally: `uv run python -m wowkb.cdmp check`.

The point of `probe guide` is **timing** — it tells you the capture is incomplete
while you are still standing at the dummy, instead of an hour later from `check`.
If it won't go green, note *which* goal is stuck in §3; that is itself the finding.

### 2.1 The WARN is two different things wearing one label

the Blizzard static spell API 404s on both, as it does for pet abilities.

  no release.**
- **`132411` = Singe Magic — REAL GAP, and it is the M3c-b defect returning.**
  Not mapped anywhere in the addon (`grep` finds nothing). Observed as
  `base=1276452 -> over=132411` — 4 events plus a live divergence, i.e. the Imp's

**Design smell this exposes.** The baseline's `known` list is a hand-maintained
*duplicate* of the addon's spec table, while its own `desc` reads "resolves to a

drift surfaces as a false WARN that costs real triage time — it just did. Candidate
follow-up: have `wowkb.cdmp` parse the ID set out of `SpecDemonology.lua` and drop
the duplicated list, so the assumption checks what it claims to check.

### 4.5 The heuristic is wonky — nothing is green, something is whiteish

**→ TWO SEPARATE DEFECTS, and the screenshot proves the bigger one is a RENDER bug,
not a heuristic bug.** Player question: *"I'm not even sure what white means?"*

**Answer: white means nothing. It is not a designed state.** The cue palette has
exactly four entries — JUDGE cyan, SOON yellow, ROTATION green, LATE green. There is
no white cue. Every white bar on screen is a cue that has **lost its colour**.

#### 4.5.a The render bug — the cue bar paints white (PRIMARY)

> **⚠ CORRECTION (player, same day).** The claim below that colour is lost for
> *every* level is **WRONG**, and the paragraph is kept rather than rewritten
> because the mistake is the instructive part. Actual report: *"The CUEs were
> usually green, they just seemed to go white periodically."*
>
> I inferred "the hue is lost for every level" from **one screenshot** and stated
> it as settled. Intermittent-white and always-white are different defects with
> different causes; a still image cannot distinguish them, and I should have said
> so instead of asserting a mechanism. **Always-white would have meant the gradient
> never works. Intermittent-white means something periodically DROPS it** — and the
> obvious periodic suspect was in front of me the whole time: `ag:Play()`, the alpha
> breathe, runs on **exactly the two levels that pulse — SOON and LATE** — while
> ROTATION, the steady green, never pulses. "Usually green, periodically white" is
> the precise signature of that. See §4.5.c.
>
> The v0.26.0 fix (base = level colour) is still right and still lands, but for a
> reason I had not identified, and it removes the *symptom* without proving the
> *cause*. That is why §4.5.c exists.

The play-test screenshot shows the summary line reading **`shards 5/5  lit 2`** with
**three** pale/white bars beside the column — i.e. 2 lit (ROTATION/LATE) + 1 JUDGE,
and **all three rendering white**. So the level hue is being lost for *every* level,
not mis-assigned for one. Colour is carrying **zero** information right now.

The mechanism is `H.paintCue` (`HudChrome.lua:409`):

```lua
local ok = pcall(function()
  t:SetColorTexture(1, 1, 1, 1)          -- base = OPAQUE WHITE
  ...
  t:SetGradient("HORIZONTAL", solid, soft)   -- hue applied ONLY here
end)
if not ok then t:SetColorTexture(c[1], c[2], c[3], a) end   -- only on a THROW
```

The bar's only source of colour is `SetGradient`. The base is white. The
correctly-coloured fallback fires **only if the pcall throws** — so if `SetGradient`
*silently no-ops* rather than erroring, `ok` is true, the fallback never runs, and the
bar stays exactly the white it was initialised to. A silent failure and a designed
state are indistinguishable on screen, which is why this survived four play-tests.

**Fix (robust regardless of why the gradient isn't taking):** make the base the level
colour, not white — `t:SetColorTexture(c[1], c[2], c[3], a)` — then apply the gradient
purely for depth. A no-opping gradient then degrades to a correctly-coloured flat bar,
and **white becomes unreachable**. I have not proven *why* `SetGradient` fails on this
client; the fix does not depend on knowing, and the assertion that it fails is
evidenced by the screenshot rather than by reading the API.

⚠ This code path is replaced wholesale by the §4.4 square-fill rework, so the fix
lands there — but the *rule* survives the rework: **never initialise a signal surface
to a colour that isn't in its palette.**

#### 4.5.c Why gradients fail *sometimes* — the watchdog (v0.27.0)

Player question: *"Is there a way we can try to figure out why gradients are
failing sometimes?"* Yes, and it is what this addon is for. `HudCueWatch` samples
`GetVertexColor` off our **own** cue textures at 4Hz, compares each against the
palette entry that cue was set to, and records every divergence with the context
needed to **test a hypothesis** rather than admire the failure: level, whether the
**alpha breathe was playing**, recede, shown, expected-vs-actual rgb. It rides
`/cdmp probe` into the structured store — evidence on disk, not in a scrollback —
and `wowkb.cdmp` asserts it as `cue-renders-its-level`.

**The prediction, stated before the data so it can be wrong:** divergences cluster
on **SOON/LATE with `playing=true`** and are **absent on ROTATION/JUDGE**. If
ROTATION diverges too, or divergences happen mostly while the breathe is *still*,
**the animation is not the cause and the hypothesis is dead** — which is worth as
much as confirming it. The check prints that split as its verdict line.

⚠ **The trap this is built to avoid:** v0.26.0 already made the base the level
colour, so a dropped gradient now degrades to a flat *correct* colour. The white
may simply stop appearing while the gradient keeps failing silently — a fix whose
mechanism was never confirmed is a fix that comes back. So the watchdog measures
`gradientLost` directly instead of trusting the absence of complaints. Default ON:
an intermittent defect is only ever caught by an instrument already running.

#### 4.5.b The churn bug — the green never sits still (SECONDARY)

Even with colour restored, the greens thrash. Quantified from the same pull, and the
player's read is right even though the on-screen `lit` meter says otherwise.

From the 15:06 pull (`CDMProbeDB.pulls`, v0.25.0, 68s tail, 153 dot transitions):

| ability | flips | times it went green | mean green dwell |
|---|---:|---:|---:|
| Demonbolt | 49 | 10 | **1.9s** |
| Hand of Gul'dan | 40 | 12 | **1.9s** |
| Shadow Bolt | 24 | 4 | 1.5s |
| Call Dreadstalkers | 11 | 1 | **0.1s** |
| Infernal Bolt | 10 | 4 | 0.8s |
| Implosion | 9 | **0** | — |
| Ruination | 6 | 1 | 3.5s |
| Summon Demonic Tyrant | 4 | 1 | 10.9s |

**2.25 dot transitions per second across the board.** Mean green dwell for the two
busiest buttons is ~1.9s — about one GCD.

**Why the histogram disagreed with the player, and why the player is right.** The
`lit` histogram for that pull reads lit=1 68%, lit=2 31%, lit=3 <1%, lit=0 **never** —
i.e. *something* is green in 100% of samples. But it is never the *same* thing for
longer than a GCD. The green strobes between buttons, so no button ever reads as
"the green one". `lit` measures how many dots are lit; it cannot see **churn**, so a
board thrashing at 2.25 Hz scores identically to a calm one. That is a gap in the
strictness meter itself, not just in the rules.

**And the whiteish thing is explained by the same data.** Implosion flipped 9 times
and went green **zero** times — it only ever reaches `AVAILABLE` /
`>=6 imps — count is secret, your call`, which paints the **cyan JUDGE** cue. So the
one *stable* cue on the board is the cyan one, while every green flickers. "Nothing
is green and something is whiteish" is a precise description of exactly this.

**Root cause — threshold chatter on the in-flight shard projection, no hysteresis.**
The transitions come in this repeating shape:

```
+179.3  Demonbolt        NEVER -> ROTATION ~est : cast in flight -> ~0 shards · core up
+180.5  Demonbolt        ROTATION -> ROTATION   : core up
+182.0  Demonbolt        ROTATION -> NEVER      : spend mode — dump shards first
+182.0  Hand of Gul'dan  NEVER -> ROTATION      : shards 4>=3
+184.3  Hand of Gul'dan  ROTATION -> NEVER ~est : cast in flight -> ~1 shards · shards 1<3
+184.3  Demonbolt        NEVER -> ROTATION ~est : cast in flight -> ~1 shards · core up
```

Every cast walks the shard total back and forth across HoG's `>=3` gate, and HoG and
Demonbolt sit on opposite sides of it, so they **trade the green cue every GCD**. The
projection (`~est`) flips the level mid-cast and the landing flips it back. Nothing
debounces this: the score is a pure function of instantaneous state, so a boundary
crossing repaints immediately in both directions.

**→ Recommended fix: hysteresis + a minimum dwell on the *cue*, not on the score.**
Keep `HudScore` pure — it's unit-tested and its purity is load-bearing — and debounce
at the paint seam: a green must persist ~250–300ms before it paints, and once painted
must hold a floor (~800ms–1 GCD) before it can go dark. Also worth suppressing the
`~est` projection from *promoting* a level at all (let it only demote), which matches
the existing napkin doctrine that an estimate may never promote a dot. Then add a
**churn** statistic to `hud log` beside `lit`, so this class of defect is measurable
next time instead of relying on the player's eye.
