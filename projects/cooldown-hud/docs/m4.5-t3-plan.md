## The pivot: why `/cdmp selftest` was shelved

The M4.5 sketch proposed a third rung — an **in-client** assertion suite
(`/cdmp selftest`) for the live Secret-Value / override / cast-readability paths
busted can't reach. On inspection that is **more machinery than the marginal value
justifies**, because **`/cdmp probe` already captures those exact observations** —
selftest would re-collect them and then bake the *expected answers* into shipped
Lua:

| selftest was going to assert | already in the probe capture |
|---|---|
| Secret API real / secret map | the secret-map section |
| combat seam (readable OOC, secret in combat) | **Section A** — cooldown readability, both contexts |
| cast readability (incl. the never-confirmed raid case) | **Section C** — per-phase cast readability |
| override resolution | **Section B** — override pairs + live base-vs-live divergence |
| scoring vs live state | the folded HUD state/score block + the `pulls` histogram |

So the missing piece was never *collection* — it is **interpretation**, which today
is a human reading `CDMProbe.lua` off disk. Move that into local tooling and every
goal selftest had is met, **better**:

- **No shipped test scaffolding** — the addon stays lean; expectations live in
  editable Python/JSON, not Lua that needs a release to change.
- **The assertions evolve without a release** — a selftest's expected answers are
  baked into the client; a local checker's are one edit away.
- **The fixture-drift risk dissolves** — nothing in-client to keep in sync with busted.
- **Richer analysis** — cross-run diffing, history, the KB — none of which
  chat-line PASS/FAIL has.

**What we give up (both minor):** an *in-the-moment* red/green in-game (we `/reload`
and check locally instead), and selftest's one non-observational trick — injecting a
controlled override and running the *real* scorer under *real* Secret Values. Since
`IsSecret`/`SpecInfo` behaviour is itself observable via probe, and the scoring logic
is busted-tested, that composition is low-risk. Neither loss is worth an in-client
suite + a per-change release.

**The rule that falls out — the whole reason this shape wins:**

> **Collect a new observation → addon change + release. Assert / interpret /
> re-verify → local, no release.** The assertions (which change often) live where
> they are cheapest to change; the addon owns only *observation* (which is stable).

## What probe does today (the starting point)

Three things, and their **output shapes differ** — which drives the one enabling
decision below:

- **Passive recorders** (from load, ~zero cost, no input): per-phase cast
  readability counters, observed override pairs, glow show/hide, CDM data-loaded,
  recent casts. Cumulative.
- **`/cdmp probe`** — the on-demand snapshot rendering everything at once: the
  passive counters + live one-shot reads (secret map, **A** cooldown readability
  per tracked spell, **B** overrides + divergence, **C** cast readability, **D**
  imp-count side channel, **E** keybind reverse index) + the HUD state/score block.
  It writes a **human-readable TEXT blob** to `CDMProbeDB.reports["probe_ooc" |
  "probe_combat"]`.
- **The pull log** (HudLog) — auto-captured on every combat exit into
  `CDMProbeDB.pulls` (last 5): lit-dot histogram, peak set + reasons, transition
  tail. Already **structured**.

Everything flushes to disk only on **`/reload`**.

## The design

### A — Addon side (one release → v0.25.0)

**A1. Structured observations output (`CDMProbeDB.probe`) — the reader's contract.**
`/cdmp probe` keeps emitting its text report for humans, but **also** persists the
same facts as a structured Lua table, because the reader must not text-parse a report
this codebase re-words freely (a maintenance trap). Keyed by combat state, mirroring
`reports`:

```lua
CDMProbeDB.probe = {
  ooc = {                                  -- a full snapshot taken at an OOC probe
    capturedAt = <GetTime()>, interface = "120007", instance = "none",
    secretAPI  = true,
    reads = {                              -- Section A, per tracked spell
      [265187] = { readable = true, duration = 0, startTime = 12345.6 },
      [196277] = { readable = false, why = "secret fields" },
      ...
    },
    divergence = { [1276467] = 388215, ... },   -- Section B live base->live
    casts   = { START={readable=N,secret=N}, SUCCEEDED={...}, STOP={...}, INTERRUPTED={...} },
    overrides = { count = N, pairs = { {base=105174, over=434635, at=...}, ... } },
    imps    = { width = 8.0, text = "3", shown = true } | nil,   -- Section D
  },
  combat = { ... same shape, taken at an in-combat probe ... },
}
```

The reader diffs `ooc` vs `combat` for `reads` exactly as a human eyeballs Section A
(the M3d combat seam). `pulls` stays as-is (already structured). One shared serializer
builds both the text lines and this table from the same in-memory observations, so the
two never diverge.

**A2. On-demand guided progress (`/cdmp probe guide`) — Option 2.** A pull-based
coverage view (no persistent frame, no auto-refresh): each **coverage goal** is a
predicate over the passive recorders, evaluated **when you type the command**. It
prints a ticked checklist + what's still missing + a nudge, and says when coverage is
complete:

```
/cdmp probe guide            (start a session with `probe clear` to reset counters)
  [✓] OOC cooldown reads captured
  [✓] SUCCEEDED seen readable
  [ ] a transform observed     <- arm a Demonic Art / let a Grimoire hit CD
  [ ] imp count >=2 seen        <- summon imps
  [ ] in-combat probe taken     <- pull a dummy, then `/cdmp probe`
  -> 2 goals left; re-run `/cdmp probe guide` to re-check, then /reload
```

You re-type it whenever you want to check "what's left." The value it buys: **you
learn a capture is incomplete while still in-game and able to fix it**, instead of an
hour later from the reader. Honest limit: it *detects and nudges* but can't *create*
state (it can't put you in a raid or force a Core proc) — it just makes the gap
visible in-game. The goal list is small and lives in the addon for the MVP; if it ever
needs to track more than a handful, mirror it from the baseline (a later change).

### B — Local side (no release, iterate freely)

**B1. The reader — a new `wowkb` module** (e.g. `wowkb.cdmp`), mirroring
`wowkb.diagnostics` one-for-one: locate the newest `CDMProbe.lua` under
`WTF/Account/*/SavedVariables/`, parse with `charstate.parse_savedvar`, read
`CDMProbeDB.probe` (+ `.pulls`). Subcommands:
- **`check`** — assert the capture against the baseline → **PASS / WARN / FAIL** per
  assumption, a **diff** vs the baseline and vs the previous capture, and a
  **"not covered this run"** list for goals the capture didn't hit. Exit non-zero on
  any FAIL (a CI-style gate you can eyeball).
- **`show`** — pretty-print the latest capture (the human render, off structured data).
- **`diff`** — capture vs capture / capture vs baseline, for spotting drift.

**B2. The baseline — JSON, patch-stamped** (`projects/cooldown-hud/probe-baseline.json`),
the addon's analogue of the KB's `verify-in-game.md`. It is the
**tested-assumptions-of-record**; each entry:

```json
{
  "patch": "12.0.7",
  "assumptions": [
    { "id": "cooldown-read-combat-seam",
      "desc": "GetSpellCooldown reads real OOC, secret in combat (M3d seam)",
      "expect": { "ooc": "all-readable", "combat": "all-secret" },
      "severity": "high", "confirmed": "2026-07-..." },
    { "id": "succeeded-readable",
      "desc": "UNIT_SPELLCAST_SUCCEEDED spellID is non-secret",
      "expect": { "casts.SUCCEEDED": "readable" },
      "severity": "high",
      "context": "confirmed delve+dummy; RAID still unconfirmed",
      "confirmed": "2026-07-..." }
  ]
}
```

The reader maps each assumption's `expect` onto a check over the structured capture.
Editing an assumption / adding a new one is a JSON edit — **no addon release**.

## The two case flows

**Case 1 — patch dropped, verify assumptions hold (no release):**
1. Patch lands; deploy the current addon (rebuild only if the interface bumped).
2. Guided capture: `/cdmp probe guide` walks the standard coverage (OOC probe →
   pull the tracked set → the raid sample if an assumption needs it → in-combat
   probe → `/reload`); the checklist confirms coverage is complete.
3. Local `wowkb.cdmp check` → parse capture, compare to baseline → **PASS** (holds)
   / **FAIL** (SUCCEEDED went secret, a spellID moved, an override changed, …).
4. Triage: fix the addon, **or** update the baseline if the game legitimately
   changed and we accept the new truth (re-stamp `confirmed` + `patch`).

**Case 2 — test a NEW assumption / extend the baseline:**
1. New hypothesis (e.g. "does `GetSpellCharges` read non-secret in combat?").
2. If probe doesn't already observe that datum → **add a passive recorder / field to
   probe (addon change + release)** — the one place a new in-game test needs a
   release, because the addon has to *observe* the new thing. Add it to `A1`'s
   structured output too.
3. Add the expectation to the baseline JSON (local, no release).
4. Guided capture → `wowkb.cdmp check` → confirm it reads as expected.
5. Promote into the baseline (`confirmed` stamped) → Case 1 now re-checks it every
   patch automatically.

## Build order

1. **A1 structured output** first — it is the reader's contract and needs the release
   anyway. Ship the shared serializer (text + table from one source).
2. **B1 reader `check` + B2 baseline** — seed the baseline from the assumptions we
   already hold (the M3d seam, SUCCEEDED readability, the confirmed override IDs, the
   expected-vs-bound set). This is where the value lands and it needs no release.
3. **A2 guided progress** — the convenience layer; ship alongside A1 in v0.25.0 (both
   are addon changes) or fast-follow.
4. Fold a one-line reader summary into `wowkb.character` / the cooldown-hud docs if
   useful; optional.

## Verification

- `wowkb.cdmp check` on a fresh capture prints PASS for every currently-held
  assumption; deliberately corrupt one field in a test capture → the matching
  assumption FAILs (prove the checker bites, mirroring T2's C1-revert).
- `/cdmp probe guide` ticks each goal as it is satisfied and reports "complete" only
  when all are met.
- A raid capture finally answers the SUCCEEDED-readability assumption's `context`
  (the standing §7 question) — the check flips it from unconfirmed to PASS/FAIL.

## Open questions / risks

- **Structured/text drift.** A1 must build both from ONE in-memory observation set
  (one serializer, two renderers) — or the text a human reads and the table the
  reader checks can disagree. Non-negotiable.
- **Baseline is only as honest as its stamps.** An assumption re-confirmed on a new
  patch must re-stamp `confirmed`/`patch`; a stale stamp silently claims a check that
  hasn't run this patch (the same failure mode the KB's `reviewed:` convention
  guards). The reader should surface each assumption's `confirmed` age.
- **Guided coverage can't force state.** `/cdmp probe guide` makes gaps visible but
  can't create a raid / a proc / imps — it nudges, the player acts. By design.
- **`pulls` vs `probe` are separate stores** with the same `/reload` flush rule; the
  reader reads both and should say when either is stale (older than the run it expects).
